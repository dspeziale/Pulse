{# ---------- 1. PANORAMICA ---------- #}

Pulse è un sistema distribuito per il monitoraggio della connettività e dello stato applicativo di sistemi che espongono un endpoint HTTP/HTTPS di heartbeat, oppure di endpoint di sola connettività (TCP host:porta).

L'architettura è composta da:

  • Server centrale: aggregazione dati, amministrazione, dashboard aggregata, gestione utenti/ruoli/permessi, notifiche e configurazione.
  • Sonde (Probe): componenti distribuiti, tipicamente vicini ai sistemi da monitorare, che interrogano periodicamente i sistemi e archiviano gli heartbeat su OpenSearch locale a ciascuna Sonda.
  • OpenSearch: la serie temporale grezza degli heartbeat risiede solo sull'OpenSearch della Sonda; il Server usa rollup aggregati e query on-demand verso le Sonde.

Tutta la comunicazione tra Server e Sonde è cifrata. Le Sonde continuano a monitorare i sistemi locali anche se il Server non è raggiungibile.

{# ---------- 2. ARCHITETTURA ---------- #}

Ruoli dei componenti:

  • Sistema monitorato: applicazione/servizio esterno; passivo rispetto a Pulse. Espone GET /api/heartbeat (tipo Applicazioni) oppure una porta TCP (tipo Connettività).
  • Sonda: interroga i sistemi a lei assegnati secondo l'intervallo di polling, indicizza ogni heartbeat come documento su OpenSearch, invia al Server liveness, eventi (cambi di stato) e rollup aggregati.
  • Server: raccoglie rollup ed eventi, calcola lo stato aggregato, genera allarmi/notifiche tramite i workflow, offre la dashboard e le pagine di amministrazione. Per il drill-down interroga on-demand l'OpenSearch della Sonda.

Flusso tipico:

Sistema monitorato → (polling della Sonda) → Heartbeat su OpenSearch locale → rollup + eventi → Server → workflow/allarmi + dashboard aggregata

Una Sonda è indipendente: il monitoraggio locale prosegue anche senza il Server.

{# ---------- 3. ACCESSO E SICUREZZA ---------- #}

  • Login con username e password. La sessione emette un token di accesso a breve durata e un refresh token; alla scadenza si viene reindirizzati al login. Dal menu Account → Profilo si cambia la propria password.
  • RBAC (Role-Based Access Control): ogni operazione è protetta da un permesso granulare (es. systems.read, probes.create). Un utente ha uno o più ruoli, ciascun ruolo raggruppa un insieme di permessi. I permessi sono un catalogo fisso definito dal sistema (non creabili a runtime). Le voci di menu e i pulsanti compaiono solo se si dispone del permesso relativo.
  • Ruoli predefiniti (es. Super Amministratore, Amministratore, Operatore, Visualizzatore, Auditor) non eliminabili + ruoli personalizzati. Deve sempre esistere almeno un Super Amministratore attivo.
  • Audit log (Sistema → Audit log): registro immutabile delle azioni che modificano lo stato (login/logout, modifiche a utenti/ruoli/ sonde/sistemi/canali/workflow/configurazione, comandi, invii notifica), con filtri.
  • Log di sistema (Sistema → Log di sistema): log tecnici dei componenti (livello, componente, sonda, messaggio, timestamp), filtrabili.
{# ---------- 4. SONDE ---------- #}

Menu Monitoraggio → Sonde. Funzioni disponibili:

  • Creazione / enrollment: creando una Sonda si ottiene un token di enrollment monouso. Va copiato subito: serve alla Sonda per registrarsi al Server e non viene più mostrato in seguito.
  • Anagrafica: oltre a nome, descrizione, endpoint di query e tag, si registrano posizione e referente (nome, email, telefono). I campi anagrafici sono opzionali.
  • Stato / liveness: ogni Sonda ha uno stato {{ status_badge('online') }} / {{ status_badge('offline') }} con ultimo contatto e versione. Il timeout oltre il quale una Sonda è considerata offline è configurabile.
  • Rotazione credenziali: dal dettaglio Sonda si possono ruotare le credenziali; viene generato un nuovo token di enrollment (anch'esso da copiare subito).
  • Drill-down: dal dettaglio Sonda si vedono i suoi sistemi, i check e gli heartbeat recenti, filtrabili per periodo/stato.
  • Pacchetto di installazione: la Sonda si installa come container Docker/Podman. Il pacchetto pronto è in deploy/probe-package/ (compose, .env.probe.example, script install.sh/ install.ps1 e INSTALL.md). Passi essenziali: compilare il file .env con endpoint del Server e token di enrollment, avviare il compose, verificare che la Sonda risulti online in questa pagina.
{# ---------- 5. SISTEMI MONITORATI ---------- #}

Menu Monitoraggio → Sistemi monitorati. L'elenco è diviso in due schede:

  • Applicazioni (tipo http): il sistema espone GET /api/heartbeat; si configura l'URL heartbeat.
  • Connettività (tipo tcp): controllo di raggiungibilità di un host:porta; si configurano host TCP e porta TCP.

Per ogni sistema si impostano: system_id (identificatore tecnico), nome, tipo, Sonda assegnata (ogni sistema appartiene a esattamente una Sonda), intervallo di polling, timeout, abilitazione e soglie di response_ms (warning ed error).

  • Testa endpoint / connessione: prima di salvare, il pulsante di test verifica la raggiungibilità del target (HTTP heartbeat o connettività TCP). Un target irraggiungibile non è un errore del test: viene riportato come non raggiungibile.
  • Check scoperti: dal dettaglio del sistema si vedono i check (check_id/check_name) rilevati automaticamente dagli heartbeat, con ultimo stato e ultimo contatto. Un sistema espone da 1 a N check.
{# ---------- 6. SCHEMA HEARTBEAT ---------- #}

Utile a chi deve esporre l'endpoint su un sistema di tipo Applicazioni. GET /api/heartbeat restituisce un oggetto (o un array di oggetti, uno per check) conforme a:

{
  "@timestamp": "2026-07-13T10:00:00Z",
  "system_id": "myapp",
  "system_name": "MyApp",
  "check_id": "db",
  "check_name": "Database",
  "status": "ok",
  "response_ms": 12,
  "message": null,
  "details": "{\"response_ms\":12,\"metrics\":{}}"
}
CampoObbligoNote
system_idObbligatorioIdentificatore tecnico del sistema.
check_idObbligatorioIdentificatore tecnico del check.
statusObbligatorioStato applicativo del check (vedi valori sotto).
@timestampConsigliatoIstante del check, ISO-8601 UTC.
system_nameConsigliatoNome leggibile del sistema.
check_nameConsigliatoNome leggibile del check.
response_msConsigliatoTempo di risposta interno del check (ms).
messageConsigliatoMessaggio descrittivo opzionale (o null).
detailsConsigliatoDettaglio strutturato serializzato come stringa JSON.

Valori di status (dominio aperto, normalizzato a):

{{ status_badge('ok') }} {{ status_badge('warn') }} {{ status_badge('error') }} {{ status_badge('down') }} {{ status_badge('unknown') }}

Valori non riconosciuti sono trattati come unknown. Se un sistema è irraggiungibile, la Sonda genera un documento con status normalizzato down.

{# ---------- 7. DASHBOARD ---------- #}

Menu Monitoraggio → Dashboard. Fornisce la vista aggregata di tutte le Sonde e i sistemi:

  • LED complessivo {{ led('ok') }} {{ led('warn') }} {{ led('error') }}: riflette lo stato dei check (systems summary). È rosso se esistono check in error o down, giallo se ci sono warn, altrimenti verde. La sua etichetta spiega il motivo con i conteggi. Non dipende dagli allarmi.
  • KPI cliccabili: totale sistemi, conteggi per stato (ok/warn/error/down/unknown), allarmi attivi, Sonde online/offline. Cliccando una tile si apre il drill-down filtrato (es. i soli check down).
  • Riepilogo per Sonda: ogni Sonda ha un proprio LED (offline → rosso; online con sistemi down → giallo; altrimenti verde), col numero di sistemi totali/down e il link al proprio dettaglio.
{# ---------- 8. QUERY E GRAFICI ---------- #}

Menu Monitoraggio → Query dati. Ricerca guidata sugli heartbeat archiviati nell'OpenSearch di una Sonda:

  • Filtri guidati a tendina concatenati: Sonda → Sistema → Check → Stato (i menu si popolano in base alla scelta precedente).
  • Preset di periodo: Ultima ora, Oggi (predefinito), Ultime 24h, 7 giorni, 30 giorni, oppure intervallo personalizzato. Il periodo è calcolato nel fuso orario configurato e convertito in UTC per la query.
  • Scorciatoia "Solo problemi" per filtrare rapidamente gli stati non ok.
  • Query avanzata: sezione collassabile con filtri/aggregazioni in formato JSON strutturato, per interrogazioni più fini.
  • I risultati sono mostrati come tabella DataTables server-side con il totale.

Menu Monitoraggio → Grafici: serie temporali del tempo di risposta (response_ms), disponibilità (uptime) e distribuzione degli stati per un sistema in un intervallo.

{# ---------- 9. NOTIFICHE E ALLARMI ---------- #}

Stato check vs Allarme. Lo stato di un check (ok/warn/error/ down/unknown) descrive il risultato del monitoraggio. Un allarme è un incidente generato da un workflow di notifica quando si verificano le sue condizioni. Sono due cose distinte: un check può essere in error senza che esista un allarme (se nessun workflow lo prevede), e viceversa.

Canali (Notifiche → Canali): Email, Telegram, WhatsApp, ciascuno coi propri parametri (es. SMTP/IMAP per Email, bot token per Telegram, provider Business API per WhatsApp). È possibile eseguire un test di invio su un canale e, dove supportato, abilitare la ricezione comandi.

Comandi in ingresso: alcuni canali possono ricevere comandi (es. /status, /ack, /silence). Telegram li supporta pienamente via webhook; WhatsApp solo con Business API; Email in modo limitato (rischio spoofing). Un comando viene eseguito solo se l'identità del canale (Notifiche → Identità canali) è associata a un utente con il permesso richiesto.

Workflow (Notifiche → Workflow): regole configurabili trigger → ambito → condizioni → controlli di soppressione → step di azione. I trigger includono cambio di stato, stato persistente, tempo di risposta superato, sistema irraggiungibile/recuperato, sonda offline/online. I controlli di soppressione comprendono throttling, deduplica, finestre di manutenzione e orari attivi. Gli step gestiscono canale/destinatari/template, ritardi ed escalation. L'invio di una notifica è governato esclusivamente da un workflow abilitato.

Allarmi (Monitoraggio → Allarmi): elenco degli incidenti; si filtrano per stato ({{ status_badge('active') }} / {{ status_badge('acknowledged') }} / {{ status_badge('resolved') }}). Un allarme attivo può essere riconosciuto (ack) dalla UI o via comando /ack: l'ack ferma le escalation successive. Al recupero del sistema l'allarme si risolve.

Storico invii (Notifiche → Storico invii): delivery log di tutte le notifiche inviate con esito, filtrabile per canale/workflow/ stato/periodo.

{# ---------- 10. CONFIGURAZIONE ---------- #}

Menu Sistema → Configurazione. I parametri globali del Server sono organizzati in schede (tab): Localizzazione, Rete & porte, Autenticazione, Sonde, Retention e Altro. Alcuni parametri, se modificati, richiedono il riavvio del servizio (indicato dopo il salvataggio).

  • Fuso orario (timezone): parametro nella scheda Localizzazione (predefinito Europe/Rome). Determina come vengono visualizzate le date-ora nell'interfaccia.
  • Normalizzazione date-ora: gli istanti sono memorizzati in UTC e convertiti nel fuso configurato al momento della visualizzazione. La stessa fonte alimenta i preset di periodo di Query dati e del Compendio.
{# ---------- 11. REPORT / COMPENDIO ---------- #}

Dal dettaglio di un sistema, il pulsante Compendio apre un riepilogo del sistema su un periodo selezionabile (predefinito: Oggi), con gli stessi preset di Query dati.

Il compendio mostra: stato complessivo nel periodo (stato peggiore + distribuzione), KPI (uptime %, response_ms medio/min/max, numero campioni, numero check, incidenti), tabella per-check e allarmi del periodo, oltre a un grafico del tempo di risposta.

Export PDF: il pulsante Scarica PDF genera lato server un report professionale (font PT Sans Narrow) con gli stessi dati del periodo scelto, pronto per l'archiviazione o la condivisione.

{# ---------- 12. TABELLE ---------- #}

Gli elenchi principali usano DataTables: ordinamento per colonna, ricerca rapida, paginazione e scelta del numero di righe per pagina. I filtri specifici della pagina (Sonda, sistema, stato, periodo…) si combinano con la ricerca della tabella.

{# ---------- 13. SCANSIONI NMAP ---------- #}

Dal menu Sicurezza → Scansioni NMAP puoi lanciare una scansione di rete eseguita da una Sonda e consultarne i risultati. Serve il permesso scans.run per avviarle e scans.read per consultarle.

Come lanciarne una

  1. Sonda: chi esegue la scansione (dalla sua rete).
  2. Target: IP, hostname o subnet CIDR (es. 10.88.2.0/24).
  3. Effort (timing): da T0 (Paranoid, lentissimo/elusivo) a T5 (Insane, rapidissimo). Default T3 (Normale).
  4. Tecnica: connect (default, senza privilegi), syn, udp, ping (solo host discovery).
  5. Porte (es. 22,80,443 o 1-1000) oppure Top porte; opzioni -sV (servizio/versione), -O (sistema operativo), -Pn (salta il ping, consigliato ON).
  6. NSE: categorie di script (default, safe, discovery, vuln, …) e/o script specifici, con eventuali argomenti; timing fine (min/max-rate, retries) e flag extra.

Privilegi

connect, -sV, ping e gli script NSE funzionano senza privilegi. Le tecniche SYN/UDP e il rilevamento OS richiedono privilegi: le capabilities del container sono gia' configurate (su Windows/Docker Desktop sono onorate in WSL2). Se mancano, la scansione riporta un errore visibile nel dettaglio.

Risultati e limiti

  • Le scansioni in corso mostrano lo stato running e la pagina di dettaglio si aggiorna automaticamente al termine (poll ~4s).
  • I risultati (host, porte, servizi/versioni, output NSE, ipotesi OS) sono salvati su OpenSearch della Sonda.
  • Limite di rete: per scansionare una LAN fisica serve una Sonda dentro la rete target.
{# ---------- 14. GATEWAY NOMINATIM ---------- #}

Il Server fa da tunnel a Nominatim (geocodifica OpenStreetMap) per le Sonde e gli altri servizi interni che non raggiungono direttamente Internet/Nominatim.

Endpoint

Base: GET /api/v1/nominatim/{search|reverse|lookup|status|details}?<query>

  • search — geocodifica diretta (indirizzo → coordinate).
  • reverse — geocodifica inversa (coordinate → indirizzo).
  • lookup / details — dettagli per identificativi OSM.
  • status — stato del servizio.

Autenticazione e limiti

  • JWT Pulse (sessione) oppure header X-API-Key (valore PULSE_NOMINATIM_API_KEY).
  • Rate-limit ~1 richiesta/secondo con cache delle risposte.

Esempio

curl -H "X-API-Key: <chiave>" \
  "http://<server>:8443/api/v1/nominatim/search?q=Roma&format=json"

Utile per la geocodifica da servizi interni (es. una Sonda che deve risolvere un indirizzo senza accesso diretto a Nominatim).

{# ---------- FAQ ---------- #}

Il LED è rosso ma gli allarmi attivi sono 0. Perché?
Il LED complessivo dipende dallo stato dei check, non dagli allarmi. Un LED rosso con 0 allarmi significa che almeno un check è in error/down, ma nessun workflow di notifica ha (ancora) generato un allarme per quella condizione.
Una Sonda risulta offline.
La Sonda non contatta il Server entro il timeout configurato. Verificare che il container della Sonda sia in esecuzione, che l'endpoint del Server e le credenziali/token nel suo .env siano corretti e che la rete tra Sonda e Server sia raggiungibile. Il timeout di offline è modificabile in Configurazione → Sonde.
Ho perso il token di enrollment.
Il token di enrollment è monouso e viene mostrato una sola volta alla creazione della Sonda. Se lo si perde, usare Ruota credenziali dal dettaglio della Sonda per generarne uno nuovo (da copiare subito).
Il test dell'endpoint dice "non raggiungibile".
Non è un errore del test: significa che il target HTTP/TCP indicato non ha risposto entro il timeout. Verificare URL/host/porta, il timeout impostato e che il sistema sia effettivamente esposto verso la Sonda assegnata.
Non vedo una voce di menu o ricevo "Permesso negato".
Le funzioni sono protette da permessi (RBAC): le voci compaiono solo se si dispone del permesso corrispondente. Richiedere a un amministratore il ruolo/permesso adeguato. Questa Guida è invece accessibile a tutti gli utenti autenticati.
Le date-ora non corrispondono al mio orario.
Gli istanti sono salvati in UTC e mostrati nel fuso orario configurato in Sistema → Configurazione → Localizzazione (predefinito Europe/Rome).
